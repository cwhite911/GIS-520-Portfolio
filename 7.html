Problem:

The problem in this exercise is to use census table data (2010 North Carolina Buncombe County census tract table T54 Race) and join it
geographic data (USA Census Tract Boundaries), and then use field T054_004 (American Indian and Alaska Native alone) to create a map of
total population of American Indian by census tract.

Analysis Procedures:

The main strategy used in the exercise is joining tables to geography. Both data sets must contain a unique identifier
field so they can be joined together in ArcMap. Sometimes the data needed to be prepared before it could be properly joined. Examples of this include
creating a new field of concatenated values or changing the data type. The other strategy used in this exercise is downloading data. Data can come from
a variety of sources, generally census.gov, but other sources are also available such as socialexplorer.com and arcgis.com.

Preparing census data for use can be troublesome, but the basic methods for taking raw census data and preparing it for GIS analysis is fairly straight
forward (Fig. 1). First, the data must be obtained from the US Census Burea or a variety of other sources. The data will be a table, ex. Census tract and
TIGER shapefile. Next, view both tables and identify which columns can be used to join the data together. Prepare each data set before the join, this can
include adding new fields, concatinating data, or changing the data type. Finally, the data can be joined together base on their unique ids

Application & Reflection:

Preparing census data for geospatial analysis is a tiresome task and can often become very confusing and complicated. The ability to properly prepare census
data is a valuable skill that all GIS analysts should have. However, the underlying skill set learned from census data is that database design, spatial or not,
needs to have fields with unique identifiers to be able to easily work together. For any project, database design should be planned from the beginning and not
as an afterthought. An example, although not as complex as census data, would be the data needed for a cemetery dataset. The project could include different
databases such as sections, lots, interments, ownership records. This data would be more valuable if they could be joined or related based on unique IDs.
